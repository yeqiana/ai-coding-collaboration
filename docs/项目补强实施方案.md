# AI Coding Guardrails 项目补强实施方案

## 目标

把当前仓库补强为可安装、可理解、可回归验证的轻量执行治理 Skill。保持最小流程原则，不把它扩张为 Agent 运行时、完整项目管理系统或 OpenSpec 的替代品。

## 现状与问题

| 问题 | 影响 | 本次处理 |
| --- | --- | --- |
| 规则分散在 Skill、规范、提示词和示例 | 内容易漂移 | 指定完整规范为唯一规范源，并用校验脚本检查关键契约 |
| 仅按复杂度分级 | 小改动也可能有高风险 | 增加风险修正：安全、权限、支付、数据和生产配置至少升级为 L2 |
| L2/L3 的需求不完整 | 直接实现易偏离意图 | 增加按需歧义追问门，不干扰 L0/L1 |
| 没有自动质量门 | 发布后才发现调用名、链接或规则缺失 | 增加无依赖 PowerShell 校验和行为契约 |
| 产品名和调用名易混淆 | 用户无法稳定调用 | 明确产品名与 Codex 稳定标识的对应关系 |

## 对标取舍

- `grill-me`：借鉴“先澄清再实现”，但只在 L2/L3 的未决决策上启用，避免小任务变重。
- OpenSpec：负责持久化的变更规格；Guardrails 负责最小改动、风险和验证证据。仅在项目已有或用户明确要求时使用。
- Superpowers：借鉴行为评测和独立验证，不复制全生命周期编排、多代理运行机制或强制 TDD。
- Hermes：属于带记忆和自动化的 Agent runtime，不纳入本 Skill 的职责范围。

## 实施内容

1. 将 `references/ai-coding-execution-spec-v1.md` 定义为规范源，`SKILL.md`、提示词与示例作为派生材料。
2. 在 `SKILL.md` 和规范源中增加风险修正、歧义追问和 OpenSpec 协作边界。
3. 统一 Codex 调用为 `$ai-coding-collaboration`，产品名保持 AI Coding Guardrails。
4. 新增 `.editorconfig`，明确 Markdown 使用 UTF-8 与 LF。
5. 新增 `evals/behavior-contracts.md`、`evals/manual-evaluation.md` 和 `scripts/test-skill.ps1`：前者定义 Agent 行为契约，人工评测执行真实 Agent，脚本只验证仓库可静态检查的契约。
6. 新增 GitHub Actions 工作流，在推送和 PR 时执行同一校验命令。

## 验收标准

| 验收项 | 证据 |
| --- | --- |
| 规则规范源明确 | 规范文件存在“唯一规范源”声明，Skill 引用该文件 |
| 高风险小改动不会按 L0 放行 | Skill、规范与行为契约均包含风险升级规则 |
| L0/L1 不被默认追问拖慢 | Skill 与行为契约明确限制歧义追问门 |
| 调用名一致 | README、中文 README、四个 prompts 均含 `$ai-coding-collaboration` |
| 静态质量门可重复执行 | Windows PowerShell 或 PowerShell 7 命令退出码为 0 |
| 行为契约可接受 | 按人工评测流程执行八个契约，保留 Agent 输出和评审结果 |

## 非目标

- 不新建 `openspec/` 或强制用户采用 OpenSpec。
- 不引入 Node、Python 或第三方测试框架。
- 不实现 Agent 记忆、多端消息、定时任务或子代理调度。

## 后续建议

当积累足够真实使用记录后，再接入具体 Agent 的自动评测 harness，记录输入、输出和评分结果；在此之前，按人工评测流程保留证据，避免把静态文本检查误报为 Agent 行为测试。
