# AI Coding 通用协作执行规范 v1.0

> 适用对象：GPT / Codex / Claude Code / Cursor Agent  
> 适用场景：软件开发、功能实现、Bug 修复、代码审查、测试验收、规范文档生成  
> 核心目标：让 AI 能根据任务大小自动选择合适流程，避免文档过多、实现漏项、测试漏项、输出过长。

## 0.1 规范层级

本文件是仓库的唯一规范源（normative source）。`SKILL.md` 是供 Agent 快速执行的摘要；`prompts/` 和 `examples/` 是辅助材料。发生冲突时以本文件为准，并在同一改动中同步修正摘要、提示词或示例。

## 0.2 Risk and ambiguity modifiers（风险与歧义修正）

任务等级按复杂度初判后，若涉及权限或安全、支付或不可逆外部动作、生产配置、数据删除/迁移/回填、公共接口或数据隔离，必须至少按 L2 处理，并在实施前说明风险和取得明确范围确认。

L2/L3 存在未决产品选择、验收标准、职责边界或不可逆取舍时，应先读取代码，再一次只提出一个聚焦问题和推荐默认项；L0/L1 不默认进入追问流程。需要持久化变更记录时，优先配合项目已有的 OpenSpec proposal/apply/archive 工作流；本规范不要求自动引入 OpenSpec。

# 0. 使用方式

把本文件放到项目根目录，例如：

```text
AI_CODING_RULES.md
```

或者合并进：

```text
PROJECT_RULES.md
AGENTS.md
CLAUDE.md
```

每次让 AI 执行任务时，可以这样说：

```text
请先阅读并遵守《AI Coding 通用协作执行规范 v1.0》。

根据任务复杂度自动判断使用 L0 / L1 / L2 / L3 哪种流程。

不要默认生成大量文档。
不要默认走完整 SOP。
必须做漏项自检。
必须输出 Executive Summary。

任务如下：
【粘贴任务】
```

# 1. 核心原则

## 1.1 人和 AI 的职责

| 角色 | 职责 |
|---|---|
| 人 | 决策、确认目标、最终验收、是否合并 |
| GPT | 需求分析、方案设计、Review、测试验收、修复提示词 |
| Codex / Claude Code | 阅读代码、修改代码、执行命令、自测、输出结果 |
| Cursor Agent | 辅助实现、局部重构、文件级修改 |

---

## 1.2 P0 绝对规则

以下规则必须遵守，违反则停止执行：

1. 复杂任务必须先分析，再给方案，最后再实现。
2. 如果用户没有明确说“实现 / 修改 / 执行”，不要直接大规模改代码。
3. 不确定时优先阅读当前项目代码、文档、配置和测试，不要凭空猜测。
4. 不允许超范围修改。
5. 不允许随意引入新框架、新依赖或大规模重构。
6. 不允许随意重命名公共接口、路由、DTO、数据库字段、配置项。
7. 不允许删除已有测试，除非明确说明原因并补充替代测试。
8. 修复问题时优先定位根因，不要只做表面绕过。
9. 如果无法验证，必须明确说明“未验证”以及原因。
10. 所有任务结束必须输出 Executive Summary。
11. 如果创建新的项目目录结构，默认增加一层 `yeqian/` 目录；已有项目禁止擅自迁移目录结构。

# 2. 任务等级矩阵

不要所有任务都走完整 SOP。  
AI 必须先判断任务复杂度，再选择对应流程。

| 等级 | 任务类型 | 示例 | 是否需要文档 | 推荐流程 |
|---|---|---|---|---|
| L0 微任务 | 5~15分钟 | 改文案、改颜色、修一个小 NPE、改日志 | 不需要 | 直接改 + 自测 + 摘要 |
| L1 小功能 | 30分钟~2小时 | 新增按钮、新增接口、简单页面、简单校验 | 只需要 Task Card | Task Card + 实施 + Checklist |
| L2 模块任务 | 半天~2天 | 登录模块、权限模块、订单模块、复杂联调 | 需要简版方案 | 简版方案 + Task 拆分 + Review + 测试 |
| L3 项目级任务 | 多天~多周 | 新项目、大重构、核心架构改造 | 需要完整文档 | PRD + 架构 + Task + Review + Release |

# 3. 各等级执行流程

## 3.1 L0 微任务流程

适用：

- 改文案
- 修简单 Bug
- 小样式调整
- 简单日志调整
- 单文件小改动

流程：

```text
理解需求
↓
直接修改
↓
自测
↓
Executive Summary
```

输出要求：

```text
【目标】
【改动文件】
【自测结果】
【风险】
【必须验收】
【结论】
```

禁止输出 PRD、架构文档、长测试方案。

---

## 3.2 L1 小功能流程

适用：

- 新增简单接口
- 新增简单页面
- 增加一个按钮或弹窗
- 增加一个表单校验
- 简单业务逻辑修改

流程：

```text
Task Card
↓
影响分析
↓
实施
↓
漏项自检
↓
P0 Checklist
↓
Executive Summary
```

必须输出一个半页以内 Task Card：

```text
任务目标：
本次范围：
明确不做：
涉及文件：
验收标准：
风险：
```

---

## 3.3 L2 模块任务流程

适用：

- 登录
- 权限
- 支付
- IM
- 文件上传
- 审批流
- 多端联调
- 涉及前后端、数据库、配置等多个层面

流程：

```text
简版方案
↓
Task 拆分
↓
逐 Task 实施
↓
Review
↓
测试
↓
漏项自检
↓
Executive Summary
```

简版方案必须控制在 500 字以内。

---

## 3.4 L3 项目级任务流程

适用：

- 从 0 到 1 新项目
- 大型重构
- 核心架构变更
- 数据库大改
- 线上发布风险高的功能

流程：

```text
PRD
↓
Architecture
↓
Task Backlog
↓
逐 Task 实施
↓
Review
↓
Test
↓
Release Checklist
↓
Knowledge 沉淀
```

L3 才需要完整文档。  
禁止把 L0 / L1 任务当 L3 处理。

---

# 4. AI 输出压缩规则

AI 默认必须简洁输出。

| 类型 | 限制 |
|---|---|
| 方案 | ≤500字 |
| Review | ≤300字 |
| 测试方案 | 使用 Checklist，不输出几十条 |
| 风险项 | 最多3条 |
| 必须验收项 | 最多5条 |
| Executive Summary | ≤300字，一屏内看完 |

如果用户明确要求详细版，才展开长文档。

---

# 5. 实施前必须做的影响分析

AI 在写代码前，必须先输出影响分析。

```text
## 影响分析

| 检查项 | 是否涉及 | 是否需要处理 | 说明 |
|---|---|---|---|
| 后端接口 | 是/否 | 是/否 |  |
| Service 逻辑 | 是/否 | 是/否 |  |
| DTO / VO | 是/否 | 是/否 |  |
| 数据库 | 是/否 | 是/否 |  |
| 配置项 | 是/否 | 是/否 |  |
| 前端页面 | 是/否 | 是/否 |  |
| 路由 | 是/否 | 是/否 |  |
| API 调用 | 是/否 | 是/否 |  |
| 权限 | 是/否 | 是/否 |  |
| 测试 | 是/否 | 是/否 |  |
```

目标：防止一开始就漏掉接口、DTO、数据库、路由、测试等关键点。

---

# 6. 最小 Diff 原则

AI 修改代码时必须遵守：

1. 修改最少文件。
2. 修改最少代码。
3. 修改最小范围。
4. 优先复用现有逻辑。
5. 不做无关格式化。
6. 不做无关重命名。
7. 不顺手重构。
8. 不顺手优化。
9. 不顺手修历史 Bug。

如果发现历史问题，记录为“后续建议”，不要混入本轮任务。

---

# 7. 实施后漏项自检

AI 写完代码后，必须进行“反向对账”。

## 7.1 需求-改动-验证对照表

```text
| 原始需求点 | 对应改动文件 | 验证方式 | 验证结果 |
|---|---|---|---|
| 需求1 | 文件A | 测试方式 | 已验证 / 未验证 |
| 需求2 | 文件B | 测试方式 | 已验证 / 未验证 |
```

要求：

1. 每个需求点必须有对应文件。
2. 每个核心改动必须有验证方式。
3. 未验证必须明确写“未验证”以及原因。
4. 没有证据不能标记为完成。

---

## 7.2 漏项自查问题

AI 必须明确回答：

```text
1. 是否有需求未实现？
2. 是否有代码改了但没验证？
3. 是否有接口、路由、DTO、配置、数据库遗漏？
4. 是否有测试未执行？
5. 当前是否可以进入 Review？
```

---

# 8. 测试分级规则

不要输出几十条测试点，统一按 P0 / P1 / P2 分级。

## 8.1 P0 必须验收

P0 是主流程和发布前必须确认的内容。

```text
☐ 核心流程可运行
☐ 数据正确
☐ 接口返回正常
☐ 页面入口存在
☐ 权限逻辑正常
☐ 构建 / 编译通过
```

## 8.2 P1 建议验收

```text
☐ 空数据
☐ 参数异常
☐ 重复提交
☐ Token 失效
☐ 网络异常
☐ 兼容旧数据
```

## 8.3 P2 可选验收

```text
☐ 大数据量
☐ 性能表现
☐ UI 细节
☐ 极端边界
☐ 弱网体验
```

---

# 9. 验收判断规则

## 9.1 可以进入下一步

满足：

```text
☐ P0 全部通过
☐ 无 Blocker
☐ 核心流程可运行
☐ 没有明确未实现需求
```

## 9.2 不允许标记为“可合并”的情况

以下情况不能写“可合并”：

1. 没有跑测试。
2. 有“未验证”项。
3. 有 Blocker。
4. 核心流程没有验证。
5. 改了接口但没有验证调用方。
6. 改了 DTO / DB / 配置但没有说明影响范围。

这种情况下只能写：

```text
可继续 / 需小修 / 必须返工
```

---

# 10. Review 输出规则

Review 只输出四部分：

```text
## 结论

可合并 / 可继续 / 需小修 / 必须返工

## Blocker

必须修复的问题。

## Major

重要问题，建议本轮修复。

## Minor

可选优化，最多5条。

## 下一步

如果需要修复，直接给出下一轮修复提示词。
```

禁止输出长篇泛泛建议。

---

# 11. Executive Summary 标准格式

每轮任务最后必须输出：

```text
【目标】
一句话说明本轮任务目标。

【改动文件】
- 文件1：改动说明
- 文件2：改动说明

【自测结果】
- 已执行命令或人工路径：结果。
- 未验证项及原因。

【风险】
- 最多3条。

【必须验收】
- [ ] 验收项1
- [ ] 验收项2
- [ ] 验收项3

【结论】
可继续 / 可合并 / 需小修 / 必须返工
```

要求：

1. 不超过 300 字。
2. 必须放在最后。
3. 风险最多 3 条。
4. 必须验收最多 5 条。
5. 如果没有测试，不能写“可合并”。

# 12. AI 生成其他规范文档时的要求

当用户要求生成规范文档时，AI 必须根据本文件生成，不要另起一套体系。

可生成的文件包括：

```text
AGENTS.md
CLAUDE.md
PROJECT_RULES.md
prompts/coding.md
prompts/review.md
prompts/bugfix.md
prompts/release.md
docs/templates/task-card.md
docs/templates/executive-summary.md
docs/templates/review-checklist.md
docs/templates/test-checklist.md
```

生成时必须遵守：

1. 不重复堆概念。
2. 不把所有流程都变成 L3。
3. 必须包含任务等级矩阵。
4. 必须包含漏项自检。
5. 必须包含 Executive Summary。
6. 必须包含“未验证不能可合并”规则。

# 13. 给 AI 生成规范文档的标准提示词

如果你要让 AI 根据本规范生成项目文件，使用：

```text
请根据《AI Coding 通用协作执行规范 v1.0》，为当前项目生成 AI Coding 规范文件。

要求：

1. 不要生成过多文档。
2. 优先生成以下文件：
   - PROJECT_RULES.md
   - AGENTS.md
   - CLAUDE.md
   - prompts/coding.md
   - prompts/review.md
   - prompts/bugfix.md
3. 每个文件只保留实际执行需要的规则。
4. 必须包含：
   - L0 / L1 / L2 / L3 任务等级矩阵
   - 最小 Diff 原则
   - 漏项自检
   - 测试 P0 / P1 / P2 分级
   - Executive Summary
   - 未验证不能标记为可合并
5. 如果文件已存在，只追加或合并，不要删除原内容。
6. 完成后输出新增/修改文件列表和 Executive Summary。
```

# 14. 给 AI 直接执行开发任务的标准提示词

如果你要让 AI 直接执行任务，使用：

```text
请先阅读并遵守《AI Coding 通用协作执行规范 v1.0》。

本任务请先判断属于 L0 / L1 / L2 / L3 哪个等级，并选择对应流程。

要求：

1. 不要默认走完整 SOP。
2. 不要默认生成大量文档。
3. 先做影响分析。
4. 遵守最小 Diff 原则。
5. 实施后必须做漏项自检。
6. 必须输出需求-改动-验证对照表。
7. 测试按 P0 / P1 / P2 分级。
8. 最后输出 Executive Summary。
9. 未验证时不能写“可合并”。

任务如下：
【粘贴任务】
```

# 15. 人工检查顺序

人不需要从头看所有输出。

推荐只看：

```text
1. Executive Summary
2. 是否存在“未验证”
3. 需求-改动-验证对照表
4. Blocker
5. P0 Checklist
```

如果这五项没问题，再看详细代码。

---

# 16. 最终目标

本规范的目标不是生成更多文档，而是：

1. 让小任务更轻。
2. 让大任务更稳。
3. 让 AI 不漏项。
4. 让人不用反复扫代码。
5. 让每次改动都有证据链。
6. 让“完成”从主观感觉变成可验证结果。
